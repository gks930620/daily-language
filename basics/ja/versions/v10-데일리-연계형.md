# v10 — 데일리 연계형

> 일본어 기초 4권 세트(①대화 입문 ②기본 문법 ③필수 표현 ④필수 한자, 6개월)의 대안 구성 버전.
> **유일하게 "종이책"이 아니다: 이 저장소의 자동 생성 파이프라인(매일 30분 페이지 + 6박스 Leitner SRS)에 기초 커리큘럼을 태우는 설계.** 비교 기준선: 정통 구성(정적인 책 4권을 각자 진도로).

---

## 1. 콘셉트

- 4권을 책이 아니라 **4개의 콘텐츠 스트림**으로 재정의한다: 매일 아침 생성되는 한 페이지가 [오늘의 대화(1권) · 오늘의 문법(2권) · 오늘의 표현(3권) · 오늘의 한자(4권) · SRS 퀴즈] 다섯 섹션으로 구성된다.
- **복습을 사람이 계획하지 않는다.** 오늘 배운 표현 4개와 한자 2자는 그날 저녁 `state/ja-basic/words.json`에 box 1로 등록되고, 이후 1·3·7·14·30·60일 간격으로 페이지 상단 퀴즈에 자동으로 돌아온다 — 이 저장소의 기존 SRS 엔진(`scripts/lib/srs.js`)을 그대로 쓴다.
- 책의 "정적인 부분"은 **커리큘럼 시드**(대화 주제 시퀀스 180일치, 문법 의존성 그래프 72항목, 표현·한자 후보 풀)뿐이다. 그날의 대화문·예문은 AI가 **"어제까지 배운 문법·어휘만 사용"** 제약 아래 매일 생성한다.
- 정통 구성 대비 본질적 차이: 진도·복습·중복 방지가 전부 **파일(state)과 코드**로 관리된다. "어디까지 했더라"가 없다 — runlog가 기억한다.

## 2. 누구에게 맞나 / 안 맞나

**맞는 사람**
- 이 저장소의 매일 페이지(en·ja-n1·ja-n2)를 이미 쓰고 있고, 같은 리듬(아침에 열어 30분)에 기초 과정을 얹고 싶은 사람.
- 복습 계획 세우기를 항상 실패하는 사람 — 이 버전에서 복습은 의지가 아니라 파이프라인이다.

**안 맞는 사람**
- 오프라인·종이 학습을 원하는 사람. 이 버전은 GitHub Pages 없이는 성립하지 않는다.
- 지금 당장 시작하고 싶은 사람 — 아래 "정직한 전제"에 쓰듯, 이 버전은 **파이프라인 확장(ja-basic 트랙 추가)이 선행되어야 하는 미래 설계**다.

### 정직한 전제 (이 버전만의 특수 조건)

- 현 시스템의 확정 철학([A6])은 "**기초는 정해진 커리큘럼으로 각자, 생성 콘텐츠는 기초 완료자용**"이다. v10은 이 경계를 옮기는 실험판이므로, 채택하려면 PLAN.md의 결정 변경이 먼저다.
- 필요한 시스템 확장(개념 수준): ① `scripts/lib/langs.js`에 `ja-basic` 트랙 등록(트랙=언어×난이도 구조에 부합) ② 시드 파일을 읽어 brief에 "오늘의 진도 + 사용 가능 문법·어휘 목록"을 주입하는 prepare 확장 ③ "배운 것만 쓰기" 제약의 검증(validate 확장 — 완전 검증은 어렵고, 최소한 문법 항목 화이트리스트 검사).
- AI가 만드는 것은 여전히 `content.json` 하나뿐이고, SRS 계산·상태 갱신은 전부 기존 결정론 코드가 한다 — 저장소의 절대 규칙을 그대로 상속한다.

## 3. 4권 구성 개요

정통 구성과의 역할 차이: 4권 = 4스트림이며, 각 스트림은 "정적 시드(사람이 한 번 설계) + 매일 생성(AI) + 자동 복습(SRS)"의 3층으로 나뉜다. 스트림마다 3층의 비율이 다르다.

| 권 | 스트림 | 정적 시드 | 매일 생성되는 것 | SRS 등록 |
|---|---|---|---|---|
| 1권 대화 | 오늘의 대화 | 주제 시퀀스 180일치(한 줄씩) | 대화문 전체(6~8줄, 배운 것만 사용) | 등록 안 함(읽기 자료) |
| 2권 문법 | 오늘의 문법 | **의존성 그래프 72항목 + 설명 원문** | 예문 3개(그날 어휘로 갱신) | 등록 안 함(참조 자료) |
| 3권 표현 | 오늘의 표현 | 후보 풀 720개(카드 원형) | 용례 문장(그날 대화와 교차) | **매일 4개 → box 1** |
| 4권 한자 | 오늘의 한자 | 빈도순 360자(카드 원형) | 용례 단어(배운 어휘 내에서) | **매일 2자 → box 1** |

**하루 페이지 구성** (`docs/ja-basic/days/<날짜>.html` — JS 0줄, `<details>`로 정답 가리기):

1. **오늘의 복습 퀴즈** — words.json에서 due인 표현·한자 카드 (자동)
2. **오늘의 대화** — 시드 주제 + 생성 대화
3. **오늘의 문법** — 시드 설명 + 생성 예문
4. **오늘의 표현 4** — 오늘 밤 SRS 등록
5. **오늘의 한자 2** — 오늘 밤 SRS 등록

### 스트림별 6개월 시드 목차

**1권 대화 스트림 — 주제 시퀀스 180일 (요약)**

| 구간 | 주제 흐름 | 한 줄 설명 |
|---|---|---|
| 1~30일 | 인사→소개→숫자→쇼핑 | 문법 G01~G12만으로 성립하는 대화 |
| 31~60일 | 주문→길→교통→시간 약속 | 존재문·시간 표현이 풀리며 대화가 길어짐 |
| 61~100일 | 일상 보고→취미→날씨→감상 | 동사문 해금 — "어제 뭐 했어요" 유형 |
| 101~140일 | 부탁→허가→전화→트러블 | て형 계열 해금, 실전 밀도 상승 |
| 141~180일 | 감정→의견→긴 대화 종합 | 배운 전부를 섞은 종합 회화, 반말 맛보기 |

**2권 문법 스트림 — 의존성 그래프 72항목 (주 3항목, 발췌)**

| 번호 | 항목 | 선행 조건(의존성) |
|---|---|---|
| G01 | AはBです | (없음 — 뿌리) |
| G02 | 〜ですか | G01 |
| G03 | 〜じゃないです | G01 |
| G04 | これ/それ/あれ | G01 |
| G05 | 〜の(소유) | G01 |
| G06 | 〜も | G01 |
| G07 | 숫자·いくら | G02 |
| G12 | 〜が あります/います | G04, G07 |
| … | … | … |
| G25 | 동사 ます형 | G01~G12 |
| G37 | て형 | G25 |
| G72 | 종합: 문장 연결 | 거의 전부 |

- 그래프인 이유: **그날 대화 생성의 제약 조건이 "오늘까지 해금된 노드 집합"으로 기계적으로 정의**되기 때문. 사람이 "이거 아직 안 배웠는데"를 감시할 필요가 없다.

**3권 표현 스트림 — 720개 (하루 4개 × 180일)**: 빈도순 + 그날 대화 주제와의 연관 우선(현 시스템의 [A7] "연관 클러스터가 잘 외워진다" 원칙 상속).

**4권 한자 스트림 — 360자 (하루 2자 × 180일)**: 빈도순, 단 그날 표현·대화에 등장하는 자를 우선 배치(교차 노출).

**시드 파일의 생김새** (개념 예시 — 사람이 한 번 만들고, prepare가 매일 읽어 brief에 주입):

```json
// seeds/dialog-topics.json (1권) — 배열 인덱스 = Day
[
  { "day": 1, "topic": "첫인사", "must_use": [], "note": "고정 정형구만" },
  { "day": 2, "topic": "이름·국적 묻기", "must_use": ["G01", "G02"] },
  { "day": 3, "topic": "직업 소개", "must_use": ["G01", "G03"] }
]

// seeds/grammar-graph.json (2권) — 노드 + 의존성
{
  "G01": { "name": "AはBです", "deps": [], "week": 1 },
  "G02": { "name": "〜ですか",  "deps": ["G01"], "week": 1 },
  "G03": { "name": "〜じゃないです", "deps": ["G01"], "week": 1 }
}
```

- prepare 확장분이 이 시드에서 "오늘의 진도"를 계산해 brief.json에 `unlocked_grammar`(해금 노드 목록)·`today_topic`을 넣으면, AI는 기존 방식대로 brief만 보고 content.json을 만든다 — **날짜·진도 계산을 AI에게 시키지 않는 현 원칙 유지.**

## 4. 1단원 샘플 — "Day 1 페이지"가 생성된 모습

> 이하는 `docs/ja-basic/days/2027-01-04.html`(가상의 Day 1)에 렌더될 내용의 마크다운 사본이다. Day 1이므로 복습 퀴즈 섹션은 비어 있다(due 항목 0) — 대신 Day 2 아침에 어떻게 보이는지를 마지막에 함께 싣는다.

### ① 오늘의 대화 (1권 스트림, Day 1: 시드 주제 "첫인사")

> 제약 조건 표시: 사용 가능 문법 = (없음 — 오늘 G01 해금 전) → **고정 정형구만으로 구성됨**

```
田中(たなか)： おはようございます。
다나카      : 안녕하세요. (아침 인사)

キム        ： おはようございます。
김          : 안녕하세요.

田中(たなか)： はじめまして。田中(たなか)です。
다나카      : 처음 뵙겠습니다. 다나카입니다.

キム        ： はじめまして。キムです。よろしくお願(ねが)いします。
김          : 처음 뵙겠습니다. 김입니다. 잘 부탁드립니다.

田中(たなか)： こちらこそ、よろしくお願(ねが)いします。
다나카      : 저야말로 잘 부탁드립니다.
```

- 읽는 법: 소리 내어 3회. 「〜です」의 정체는 바로 아래 오늘의 문법에서.
- (시스템 뒷면: 이 대화는 content.json의 `conversation`으로 생성되며, `reading` 필드에 전문 히라가나가 함께 온다 — 기존 ja 트랙과 같은 스키마.)

### ② 오늘의 문법 (2권 스트림, G01 「AはBです」)

**시드 설명(정적 — 매일 같은 원문)**

> **AはBです** — "A는 B입니다". 일본어 문장의 뿌리. 어순이 한국어와 같아서 단어만 갈아 끼우면 된다. 조사 は는 **[와]**로 읽는다.
> 의존성: 없음(첫 항목). 이 항목이 해금되면 → G02(ですか)·G03(じゃないです)이 열린다.

**생성 예문(동적 — 오늘의 대화·어휘와 교차)**

```
私(わたし)は キムです。          저는 김입니다.
田中(たなか)さんは 日本人(にほんじん)です。   다나카 씨는 일본인입니다.
私(わたし)は 韓国人(かんこくじん)です。       저는 한국인입니다.
```

### ③ 오늘의 표현 4 (3권 스트림 → 오늘 밤 SRS 등록)

| 표현 | 읽기 | 뜻 | 암기 노트 |
|---|---|---|---|
| おはようございます | おはようございます | 안녕하세요(아침) | 낮 인사는 こんにちは, 밤은 こんばんは — 세트로 기억 |
| はじめまして | はじめまして | 처음 뵙겠습니다 | 初(はじ)めて(처음으로)와 같은 뿌리 |
| よろしくお願(ねが)いします | よろしくおねがいします | 잘 부탁드립니다 | 願(ねが)う(바라다)가 속에 있다 — "잘 되길 바랍니다"의 정형화 |
| ありがとうございます | ありがとうございます | 감사합니다 | 장음 두 곳([아리가토-고자이마스]) — 짧게 끊으면 어색 |

**시스템 뒷면 — 카드가 상태 파일에 들어가는 모습** (settle 후 `state/ja-basic/words.json`, 기존 스키마 그대로):

```json
"よろしくおねがいします": {
  "added_on": "2027-01-04",
  "box": 1,
  "next_due": "2027-01-05",
  "graduated": false,
  "history": [{ "date": "2027-01-04", "event": "added" }],
  "card": {
    "pos": "표현",
    "reading": "よろしくおねがいします",
    "ko": "잘 부탁드립니다",
    "note": "願(ねが)う(바라다)가 속에 있는 정형 인사",
    "related": [{ "word": "こちらこそ", "note": "받아치는 짝 표현", "ko": "저야말로" }]
  }
}
```

- `note`·`related`는 복습 퀴즈의 정답 면에 매번 재노출된다(현 시스템 [A7] 그대로) — 4번째 복습쯤엔 짝 표현까지 함께 굳는다.

### ④ 오늘의 한자 2 (4권 스트림 → 오늘 밤 SRS 등록)

| 자 | 훈독 | 음독 | 용례(배운 어휘 안에서) | 암기 노트 |
|---|---|---|---|---|
| 日 | ひ | ニチ | 日本(にほん) 일본 | 한국음 "일"과 음독 ニチ는 사촌. 해 모양의 상형 |
| 本 | もと | ホン | 日本(にほん) 일본 / 本(ほん) 책 | 나무(木) 뿌리에 획 하나 = 근본. 한국음 "본"↔ホン |

- 오늘의 대화에 나온 단어(日本人(にほんじん))에서 두 자를 뽑았다 — 한자 스트림이 대화 스트림을 뒤따라가며 교차 노출하는 설계.
- 한자 카드도 같은 words.json에 들어간다: `headword: "日"`, `card.reading: "ひ・ニチ"`, `card.ko: "날 일"` — 스키마 확장 없이 "reading 있으면 렌더" 규칙으로 처리.

### ⑤ Day 2 아침의 복습 퀴즈 (참고 — 자동으로 이렇게 돌아온다)

> Day 1의 6항목이 전부 due(box 1, next_due = +1일). `<details>`를 눌러야 정답이 보인다.

```
Q1. [표현] "잘 부탁드립니다" 를 일본어로?
    ▸ (펼치면) よろしくお願(ねが)いします
      note: 願(ねが)う(바라다)가 속에 있는 정형 인사 / 짝 표현: こちらこそ

Q2. [한자] 日 — 읽는 법 두 가지와 뜻은?
    ▸ (펼치면) ひ・ニチ / 날 일 / 日本(にほん)

Q3. [표현] 아침 인사 "안녕하세요"는?
    ▸ (펼치면) おはようございます
      note: 낮 こんにちは / 밤 こんばんは 와 세트

Q4. [한자] 本 — 훈독·음독과 용례는?
    ▸ (펼치면) もと・ホン / 근본 본 / 日本(にほん)・本(ほん) 책

Q5. [표현] "처음 뵙겠습니다"는?
    ▸ (펼치면) はじめまして — 初(はじ)めて(처음으로)와 같은 뿌리

Q6. [표현] "감사합니다"는? (장음 위치 주의)
    ▸ (펼치면) ありがとうございます [아리가토- 고자이마스]
```

- 전부 맞히면(현 v1 규칙: 노출=통과) 6항목이 box 2로 승급, 다음 재등장은 3일 뒤. 이후 7·14·30·60일 — **사람은 아무것도 계획하지 않았다.**

### 시스템 뒷면 — ja-basic 트랙의 하루 (기존 파이프라인 그대로)

| 단계 | 하는 일 (v10에서 달라지는 부분만 굵게) |
|---|---|
| prepare | due 카드 동결(review.json) + **시드에서 오늘의 진도 계산 → brief에 unlocked_grammar·today_topic 주입** |
| generate (AI) | brief를 보고 content.json 하나 생성 — 대화·문법 예문·표현 4·한자 2 |
| settle | 검증(**문법 화이트리스트 검사 포함**) → 표현·한자를 box 1 등록 → due 승급 → selected.json 동결 |
| build | docs/ja-basic/ 페이지 재생성 (전 트랙 공통 빌더) |

### 부록 — 정통 구성(정적 책 4권)과의 차이 한눈에

| 비교 축 | 정통 구성 | v10 데일리 연계형 |
|---|---|---|
| 진도 관리 | 사람이 책갈피로 | runlog + 시드 인덱스가 자동으로 |
| 복습 계획 | 사람이 주말에 몰아서 | SRS가 매일 아침 due를 산출 (의지 개입 0) |
| 콘텐츠 신선도 | 인쇄 시점에 고정 | 예문·대화가 매일 새로 생성(어제 배운 어휘를 오늘 재사용) |
| 중복·누락 방지 | 없음(사람 몫) | settle의 dedup + selected.json 동결 |
| 놓친 날 | 밀린 페이지가 부채로 쌓임 | 그날은 그냥 사라지고 due만 늘어남 — 부채 없음 |
| 필요 인프라 | 책 한 질 | GitHub Actions + Pages + 트랙 확장 구현 |
| 실패 모드 | 의지 소진 | 생성 품질 저하·파이프라인 장애 (verify가 커밋을 막아 최소한 깨진 페이지는 안 나감) |

**정직한 리스크 요약**: ① **생성 품질 변동** — "배운 것만 사용" 제약을 AI가 어길 수 있다. validate의 문법 화이트리스트 검사로 걸러도 어휘 누출은 완전 차단이 어렵다(정통 구성엔 없는 리스크). ② **시스템 선행 투자** — ja-basic 트랙 추가·prepare 시드 주입·validate 확장이 되기 전엔 이 버전은 종이 위의 설계다. 또한 [A6] 철학의 공식 변경(PLAN 수정)이 필요하다. ③ **놓친 날의 처리**는 오히려 강점 — runlog 멱등 키 구조상 밀린 날은 그냥 사라지고 다음 날 due가 늘어날 뿐, "따라잡기 부채"가 진도를 무너뜨리지 않는다(스파르타형 v09와 정반대의 성질). ④ 오프라인 불가·저작 시점에 존재하지 않는 트랙이라는 점은 위 "정직한 전제" 참조.
